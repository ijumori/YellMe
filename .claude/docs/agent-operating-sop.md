# YellMe エージェント運営SOP

## 1. 1チケット運営フロー（標準）
1. Chiefが対象チケットを選定（P0優先）
2. PM Agentが受け入れ条件を3-5項目で固定
3. Core/UI Dev Agentが実装
4. QA/Test Agentが正常系・異常系を確認
5. Safety Agentが認可/機密/Rulesを監査
6. Chiefが統合判断してコミット可否を決定
7. Release Agentがノートと運用記録を更新

## 2. ダブルチェック項目（必須）

### A. 実装ダブルチェック（Dev以外）
- 仕様どおりの導線か
- 既存機能の退行がないか
- 例外・エラー時にUIが壊れないか

### B. テストダブルチェック（QA + Chief）
- 主要導線を最低1本、Chiefが再実行
- QA観点漏れ（境界値/ネットワーク断/未ログイン）確認

### C. セキュリティダブルチェック（Safety）
- 認可: `request.auth.uid` と対象 `userId` の整合
- Rules: Firestore/Storageが個人データを保護
- 機密: APIキーや証明書がコミット対象にない

## 3. Definition of Done（DoD）
- 受け入れ条件をすべて満たす
- Lintエラー0
- QA確認結果が記録されている
- Safetyチェック完了
- 関連ドキュメント更新済み

## 4. 不測事態ハンドリング

### P0障害（クラッシュ/データ欠損/認可破綻）
- 直ちに新規開発停止
- Chiefが影響範囲を15分以内に確定
- Hotfixかロールバックかを30分以内に決定

### ロールバックルール
- 直前安定コミットIDをRelease Agentが管理
- 影響説明と復旧計画を同時に記録

## 5. 日次運用
- 朝会（15分）: 昨日の完了、今日の着手、ブロッカー
- 夕会（10分）: チケット進捗、未解決リスク
- 週次（30分）: KPI確認と次週優先度更新
